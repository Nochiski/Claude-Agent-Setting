# PR 작성 워크플로우

AI 코드를 프로덕션에 안전하게 통합하기 위한 PR 계획-실행 워크플로우입니다.

> **핵심 원칙**: "신뢰하되 검증 (proof over vibes)"

---

## Phase 1: 계획 수립

### 1. 변경 범위 정의
- 이 PR의 목적은 무엇인가요?
- 어떤 문제를 해결하나요?
- 어떤 기능을 추가하나요?

### 2. 영향 분석
다음을 탐색하세요:
- 수정할 파일 목록
- 영향받는 기존 기능
- 필요한 테스트

### 3. PR 계획 작성 (4요소 기반)

```markdown
## PR 계획

### 제목
[conventional commit 형식: feat|fix|refactor|docs|chore]

### 의도 (Intent)
- 무엇을 변경하나요?
- 왜 이 변경이 필요한가요?

### AI 역할 (AI Contribution)
- [ ] AI가 생성한 코드 비율: ___%
- AI가 작성한 주요 부분:
  - `파일:라인` - 설명

### 변경 파일
1. `path/to/file.ts` - 변경 내용
2. ...

### 위험도 (Risk Level)
- [ ] 낮음: 단순 버그픽스, 리팩토링
- [ ] 중간: 기능 추가, 로직 변경
- [ ] 높음: 아키텍처 변경, 보안 관련

### 리뷰 초점 (Review Focus)
> 인간 리뷰어가 특히 집중해야 할 1-2개 영역:
1.
2.

### 테스트 계획
- [ ] 테스트 항목 1
- [ ] 테스트 항목 2

### 작동 증거 (Proof of Work)
- [ ] 테스트 통과 스크린샷/로그
- [ ] 로컬 동작 확인
```

### 4. 계획 확인
"이 계획으로 PR을 작성할까요?"

---

## Phase 2: 실행

계획 승인 후:

1. **Auto-accept 모드 권장**: `Shift+Tab`으로 전환
2. 계획대로 순차 구현
3. 각 단계 완료 시 체크
4. **필수 파이프라인 실행**: code-reviewer → test-writer → formatter
5. 모든 변경 완료 후 PR 생성

---

## PR 4요소 체크리스트

| 요소 | 설명 | 체크 |
|------|------|------|
| **의도 명시** | 변경의 무엇과 왜를 1-2문장으로 | [ ] |
| **작동 증거** | 테스트 통과, 스크린샷, 로그 등 | [ ] |
| **AI 역할** | AI가 생성한 부분 명시 | [ ] |
| **리뷰 초점** | 인간 검토가 필요한 1-2개 영역 | [ ] |

---

**Tip**: 설명할 수 없는 코드는 절대 커밋하지 마세요.
